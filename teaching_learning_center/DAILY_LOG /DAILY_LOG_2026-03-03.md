### 补充学习记录（InterpoNet / ONNX / TensorRT 概念理顺）

- 这次真正理顺的 5 层关系：
  1. `external/InterpoNet/model.py`
     - 定义原始神经网络结构（真正的前向图）
  2. `TF checkpoint`
     - 保存训练得到的参数，不是单文件，通常由 `.meta / .index / .data-*` 组成
  3. `external/InterpoNet/export_interponet_onnx.py`
     - 负责把“网络结构 + checkpoint 参数”导出成 `ONNX`
  4. `TensorRT engine`
     - 由 `ONNX` 再编译出的 GPU 推理产物，和 TensorRT/GPU/shape profile 强相关
  5. `src/inference/InterpoNetEngineTRT.cpp`
     - 不是网络结构定义文件，而是 C++ 侧的“输入构造 + engine 调用 + 输出后处理”包装层

- 关键认知更正：
  - `ONNX` 不是把整个 Python `InterpoNet.py` 原封不动打包
  - `ONNX/engine` 只包含网络前向图本体，不包含：
    - 读取 matches / edges
    - `downscale_all`
    - `BA matches` 融合
    - 上采样
    - variational 后处理
  - 所以上述外围语义必须由 `InterpoNetEngineTRT.cpp` 在 C++ 里复刻，否则即使 TRT 推理数值稳定，端到端精度也可能偏差

- 这次额外理解到的部署价值：
  - TensorRT 的核心不是“把 Python 变成 C++”，而是把已经训练好的模型前向图转成更适合 NVIDIA GPU 推理的执行形式
  - 这意味着很多原本在 Python 框架中训练/验证的模型，都可以迁移到 TensorRT 做部署加速
  - 迁移后的典型收益：
    - 更低延迟
    - 更高吞吐
    - 更小显存/内存占用（取决于优化与精度模式）
    - 更容易接入 C++ 服务或单进程高性能流水线

- 对“Python 模型 vs TRT 推理模型”的可比性理解：
  - 两者可比较，但比较的是“同一模型在不同部署形态下的推理效果”
  - 正确比较前提不是只看输出速度，而是先确认：
    1. 输入预处理是否一致
    2. 输出后处理是否一致
    3. 使用的 checkpoint / onnx / engine 是否同源
  - 如果这些条件不满足，看到的差异不一定来自 TRT 本身，而可能来自外围语义不一致

- 回流到当前项目的最重要启发：
  - 我们现在卡住的精度问题，不一定是 `TRT engine` 本体错误
  - 更可能是 `InterpoNet` 原始 Python 流程中某些“网络外语义”还没有被 C++ 版完整复现
