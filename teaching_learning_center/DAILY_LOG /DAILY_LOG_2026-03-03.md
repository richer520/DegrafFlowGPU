## 今日单点优化（最多 1-2 个）

### 优化 1（RAFT 后端切换）
- 目标文件/函数：`src/inference/RaftEngineTRT.cpp::estimateMatchesBatch`
- 问题（为什么要改）：当前 RAFT 路径长期停留在 LK fallback，无法达成真 TRT 推理目标
- 动作（只改了什么）：新增 `DEGRAF_RAFT_BACKEND / DEGRAF_RAFT_ENGINE_PATH / DEGRAF_ALLOW_LK_FALLBACK` 控制；默认走 `trt`，仅显式允许时回退 `lk`
- 结果（指标变化）：后端控制清晰，避免“名为TRT实为LK”的隐性偏差
- 风险（潜在副作用）：未提供 engine 或 TensorRT 编译支持时会失败（严格模式预期行为）

### 优化 2（TensorRT 推理骨架）
- 目标文件/函数：`src/inference/RaftEngineTRT.cpp`
- 问题（为什么要改）：需要从 engine 直接推理并输出 matches，才能形成 C++ 端到端链路
- 动作（只改了什么）：新增 TensorRT engine 加载、前向执行、dense flow 到 sparse matches 采样；失败时按策略回退 LK
- 结果（指标变化）：具备“读取 `.engine` -> 推理 -> 匹配输出”的主干能力
- 风险（潜在副作用）：不同 RAFT engine 的输入输出 binding 名可能不一致，需要按实际 engine 校验

### 优化 3（误差定位导出与对齐修正）
- 目标文件/函数：`src/inference/RaftEngineTRT.cpp::inferDenseFlow / estimateMatchesBatch`
- 问题（为什么要改）：当前精度异常，需要快速确认误差来自 RAFT 对接还是 EPIC 后处理
- 动作（只改了什么）：
  - 将 padding 修正为对齐 KITTI 数据集的 Python `InputPadder(mode='kitti')` 语义
  - 保留 padded flow 分辨率用于稀疏采样，避免与 Python RAFT 的采样行为不一致
  - 新增 `DEGRAF_DEBUG_DUMP_MATCHES_DIR` 导出开关，按帧导出 `cpp_trt_matches_frame_xxxxxx.txt`
  - 新增 `external/RAFT/debug_dump_matches.py` 与 `scripts/compare_sparse_matches.py`，用于一帧基线对比
- 结果（指标变化）：可直接做 `cpp_trt vs python_raft` 一帧误差对比，快速定位“模型输出/坐标映射/符号方向”哪一层出错
- 风险（潜在副作用）：若点序或过滤规则不一致，会出现样本数不同，需要按同一帧同一点集比较

### 优化 4（RAFT 真实一致性验收）
- 目标文件/函数：`src/inference/RaftEngineTRT.cpp`、`external/RAFT/debug_dump_matches.py`
- 问题（为什么要改）：要确认 `RAFT(TRT)` 不是“能跑”而是“与 Python 语义一致”
- 动作（只改了什么）：完成同帧同点集对齐，修复 compare 口径，重新导出/构建 FP32 engine 并复测
- 结果（指标变化）：`cpp_trt` vs `python_raft` 稀疏匹配误差降到 `~1e-3`（近似一致）
- 风险（潜在副作用）：若后续更换 ONNX/engine，需重复一次一帧对齐验收

### 优化 5（InterpoNet 真 ONNX/TRT 并入）
- 目标文件/函数：`external/InterpoNet/export_interponet_onnx.py`、`src/inference/InterpoNetEngineTRT.cpp`
- 问题（为什么要改）：原 InterpoNet 导出脚本是占位，无法真正进入 in-process TRT
- 动作（只改了什么）：补齐 TF ckpt -> ONNX 导出链路；新增 InterpoNet TRT backend，支持 `image/mask/edges` 三输入推理
- 结果（指标变化）：完成 `interponet_kitti.onnx` 与 `interponet_kitti_fp32.engine` 构建并在 C++ 路径运行
- 风险（潜在副作用）：TF1/TF2 兼容依赖复杂，云端环境变更时可能需重新校准依赖

### 优化 6（InterpoNet 输入语义对齐 + SED 边缘接回）
- 目标文件/函数：`src/inference/InterpoNetEngineTRT.cpp`
- 问题（为什么要改）：TRT 端初版输入用简化构造，精度明显低于预期
- 动作（只改了什么）：
  - 下采样改为贴近 Python `downscale_all` 的 block-nanmean 语义
  - 优先读取 `edges.dat`（SED），缺失才回退 Canny
  - 增加每帧日志 `edges_source=sed_bin/canny`
- 结果（指标变化）：确认运行已使用 `edges_source=sed_bin`
- 风险（潜在副作用）：若 `edges.dat` 帧号与评测帧不对齐，会出现隐性偏差

### 优化 7（Dense 级别 C++/Python 一致性验收）
- 目标文件/函数：`src/inference/InterpoNetEngineTRT.cpp`（dense dump 开关）
- 问题（为什么要改）：需要确认 InterpoNet TRT dense 输出是否接近 Python dense 输出
- 动作（只改了什么）：新增 `DEGRAF_INTERPONET_DEBUG_DUMP_DIR` 导出 `*_trt_dense.flo`，与 Python 版本同帧比较
- 结果（指标变化）：
  - `000000`: `mean_abs_dx≈0.104` `mean_abs_dy≈0.040` `mean_epe≈0.120`
  - `000001`: `mean_abs_dx≈0.103` `mean_abs_dy≈0.040` `mean_epe≈0.117`
  - 结论：InterpoNet TRT dense 与 Python dense 已较高一致
- 风险（潜在副作用）：最终 SceneFlow 指标仍可能受 variational/评估口径影响
